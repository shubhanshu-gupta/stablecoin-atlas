
STABLECOIN ATLAS
Product Requirements Document
Feature: Stablecoin Market Dashboard
Homepage Hook  ·  v1.0  ·  For Antigravity  ·  March 2026


This PRD specifies the Stablecoin Market Dashboard — a live data panel that lives on the homepage of stablecoinatlas.app. It is the platform's 'daily hook': a reason for professionals, learners, and casual observers to return every day. All data is sourced from free, no-auth-required public APIs (primarily DeFiLlama Stablecoins API). No paid data subscriptions are required for MVP.

1. Overview

1.1  Problem Statement
The stablecoinatlas.app homepage currently shows a static hero section with three navigation links. There is no live data, no reason to return daily, and no immediate 'wow' moment for a first-time visitor. A professional who lands on the page for the first time sees a marketing page — not a tool.

The homepage needs a hook: something that communicates, in under five seconds, that this is the most useful single place to understand the stablecoin ecosystem right now.

1.2  Solution
A live Stablecoin Market Dashboard embedded directly on the homepage, above the fold on desktop and as the first scroll section on mobile. It surfaces the metrics that stablecoin professionals actually care about — total supply, supply trends, issuer dominance, chain distribution, and peg health — all updating in real time from DeFiLlama's free public API.

Design reference: Think Bloomberg terminal meets a clean fintech dashboard. Data-dense enough to be useful to a compliance officer or product manager. Clean enough that a curious newcomer can understand it without explanation.

1.3  Goals
PRIMARY GOALS
	•	Give every visitor — expert or newcomer — an immediate, live signal about the stablecoin market.
	•	Create a daily return habit: professionals check the dashboard like they check Bloomberg or Dune.
	•	Establish Stablecoin Atlas as a credible, data-driven platform, not just an editorial site.
SECONDARY GOALS
	•	Drive deeper engagement: each metric should link to a relevant page (Learn, Latest, entity pages).
	•	SEO benefit: live data pages earn backlinks and rank for high-intent queries like 'stablecoin market cap'.

1.4  Non-Goals (MVP)
	•	No trading functionality, price prediction, or yield data.
	•	No user accounts or personalised watchlists (v2).
	•	No push/email alerts for market movements (v2).
	•	No historical chart builder or custom date ranges on the homepage (v2 — link to dedicated /dashboard).
	•	No paid data sources required for MVP.

3. Data Sources & API Specification

3.1  Primary Source: DeFiLlama Stablecoins API
DeFiLlama's Stablecoins API is free, requires no API key, and is the most comprehensive open source of stablecoin on-chain supply data. It is used by The Block, Messari, and major crypto research firms. Attribution is required (DeFiLlama citation in dashboard footer).

Base URL: https://stablecoins.llama.fi  ·  Rate limit: ~300 req/min (free tier)  ·  No API key required  ·  CORS-friendly (can call from frontend or backend)

3.3  Data Caching Strategy
Raw API calls must NOT be made from the browser on every page load. The following caching architecture is required:

	•	A server-side cron job (or Vercel/Netlify scheduled function) fetches from DeFiLlama every 60 minutes and stores the response in a lightweight cache (Redis, KV store, or even a JSON file in a CDN edge cache).
	•	The frontend requests data from this cached layer (/api/dashboard), not directly from DeFiLlama.
	•	Cache TTL: 60 minutes for supply data. 24 hours for historical chart data (it doesn't change intraday).
	•	Show a 'Last updated: X minutes ago' timestamp on the dashboard so users know data freshness.
	•	Stale-while-revalidate pattern: serve cached data instantly, refresh in background.

3.4  Supplementary Source: CoinGecko (optional, v1.1)
For 24h trading volume and exchange data per stablecoin, CoinGecko's free tier can be used (50 calls/minute, no auth required). This is not required for MVP but can be layered in for the leaderboard's 'Volume' column in v1.1.

Module order (top to bottom / left to right): [1] Headline KPIs → [2] Supply Trend Chart → [3] Issuer Leaderboard → [4] Chain Distribution → [5] Peg Health Monitor.

MODULE 2  —  Total Supply Trend Chart

Purpose
An area/line chart showing total stablecoin market cap over time. This gives context to the headline KPI — is the market growing, contracting, or recovering? The 2022 UST crash, 2023 BUSD de-listing, and 2024 bull market growth should be visible and labelled.

Chart specification
	•	Chart type: area chart with a gradient fill (blue at line, fading to transparent at baseline).
	•	X-axis: time (daily data points). Default view: 1 year. Toggle buttons for: 3M / 6M / 1Y / All time.
	•	Y-axis: USD billions, auto-scaled.
	•	Hover tooltip: exact date + total supply in $B + % from previous period.
	•	Annotations: 2–3 key events marked with a vertical dashed line and label (e.g. 'UST Collapse — May 2022', 'BUSD Wind-down — Feb 2023'). These are hard-coded editorial annotations, not dynamic.
	•	Data source: GET /stablecoincharts/all — returns array of {date, totalCirculatingUSD} objects.
	•	Library recommendation: Recharts (if React) or Chart.js. Both are lightweight and handle time-series well.

MODULE 4  —  Chain Distribution

Purpose
Shows how stablecoin supply is distributed across blockchain networks. This matters to product managers, engineers, and BD teams who need to know where liquidity actually lives. It is also the clearest indicator of which chains are 'winning' as stablecoin rails.

Visualisation specification
	•	Primary: horizontal bar chart (easier to read chain names than a donut on mobile).
	•	Secondary toggle: donut chart view (for users who want relative proportions).
	•	Show top 8 chains by supply. Remaining chains are grouped as 'Others (N chains)'.
	•	Each bar: chain name (left label), supply in $B (right label), percentage of total (inside bar or tooltip).
	•	Colour coding: one distinct colour per chain (Ethereum = purple, Tron = red, Solana = green, etc.). Consistent with any future entity pages.
	•	Data source: GET /stablecoinchains — returns array of {name, totalCirculatingUSD}.
	•	Clicking a chain bar: links to a future /chains/[chain-name] entity page. For MVP, opens a DeFiLlama chain page in a new tab.

MODULE 5  —  Peg Health Monitor

Purpose
Shows whether major stablecoins are trading at their intended peg. A stablecoin at $0.97 USD is significant news. This module makes Stablecoin Atlas uniquely useful in moments of market stress — when it matters most, professionals will check here first.

What is shown
	•	Top 10 stablecoins by supply, each showing: current price, deviation from peg (in cents and percentage), status badge.
	•	Status badges: ON PEG (within ±0.5%) shown in green, WATCH (±0.5–1.0%) in amber, DEPEG ALERT (>1.0%) in red.
	•	Price data: sourced from /stablecoins?includePrices=true — the price field per stablecoin.
	•	'Last updated' timestamp is shown prominently. Refreshes every 60 minutes.

Depeg alert behaviour
	•	If any stablecoin in the top 10 shows >1% deviation: the module header colour changes from neutral to red, and a banner appears: '[Coin name] is showing a peg deviation of X%. Monitor closely.'
	•	No push notifications in MVP. Alert is visual only.
	•	The banner links to the Latest section filtered for the relevant stablecoin — so users can see if there's news context.
Important: include a disclaimer inline: 'Price data is sourced from DeFiLlama and reflects on-chain oracle prices. It may differ from centralised exchange prices. This is not financial advice.'

6. Technical Requirements

6.1  Data Pipeline
	•	Create a server-side endpoint: GET /api/dashboard — returns all dashboard data in one payload.
	•	This endpoint fetches from DeFiLlama on a schedule (cron or on-demand with cache-first strategy).
	•	Response shape: { kpis: {...}, supplyHistory: [...], leaderboard: [...], chainDistribution: [...], pegHealth: [...], lastUpdatedAt: timestamp }
	•	Cache the full response for 60 minutes. Use stale-while-revalidate: return cached data instantly, refresh in background.
	•	The frontend calls /api/dashboard once on page load. No individual component-level API calls.
